# 07. Tauri 재구현 고려사항

> 일부는 확정, 일부는 미정. 아래 "확정 결정"과 "열린 질문"을 구분해서 볼 것.
> (최종 갱신: 2026-06-11)

## 왜 Tauri로 옮기는가 (배경)

기존 Next.js + Vercel + Supabase 구성에서 겪은 문제:

1. **Vercel 서버리스 불안정**: 배포 환경/API가 계속 바뀌어 원활하게 운영하기 어려웠음.
2. **Supabase 무료티어 일시정지**: 사용량이 적으면 프로젝트가 멈춰버려 무료로 안정적 운영이 안 됨.

→ **Tauri 데스크톱 앱 + 앱 내장 로컬 SQLite** 로 전환하면, 서버·외부 DB 의존이 사라지고
각 사용자의 PC 안에 데이터가 쌓이므로 비용·정지 문제가 없어진다. 1인·로컬 도구 성격에 부합.

## ✅ 확정 결정

| 항목 | 결정 | 비고 |
|------|------|------|
| DB | **로컬 SQLite** (앱 빌드에 내장) | Supabase 폐기. 사용자별 자기 PC에 데이터 저장 |
| secret key / 설정 저장 | **로컬 저장** | 쿠팡 키가 서버·브라우저에 노출되지 않음 |
| 대상 OS | **macOS + Windows** 크로스플랫폼 | 개발은 Mac, 사용자는 Windows도 포함 |
| 경로 처리 | **Tauri path API(`app_data_dir` 등) 필수** | 경로 하드코딩 금지 (Mac/Win 경로 상이) |
| 로그인/인증 | **제거** | 공개 URL이 없는 로컬 앱 + OS 계정이 이미 게이트. `admin_password`·미들웨어·login 페이지 폐기 |
| 5초 블로킹 대기 | **기존 그대로 유지** | 등록/수정 후 5초 대기 후 상태 조회. 추후 개선 여지만 남김 |
| 프론트엔드 | **기존 React 컴포넌트 재사용** | Vite로 이식, 나중에 수정 |
| 크롤링 방식 | **기존 HTTP 파싱 그대로** | hankyeong.kr 셀렉터 맵 재사용 |
| 카페 사이트(`/(main)`) | **제외** | 쿠팡 어드민 기능만 포팅 |

### 로그인 제거 → "초기 설정"으로 대체
- 로그인은 없애지만, 각 사용자가 **자기 쿠팡 API 키 + 셀러 기본정보(반품지/출고지 등)를 입력하는
  설정(온보딩) 화면**은 필요하다. (예전엔 Vercel 환경변수에 있던 것을 이제 앱에서 직접 입력)
- 즉 "인증 비밀번호"는 사라지고, **"쿠팡 키 입력 설정"만 남는다.** 빈 DB로 배포되며 첫 실행 시 사용자가 본인 키를 입력해 로컬에 저장.

### 확정에 따른 제약 (수용함)
- 데이터는 **기기 로컬에만** 존재 → 기기 간 동기화 없음.
- **백업은 사용자 책임** → 추후 "DB 내보내기/가져오기" 기능으로 보완 예정.
- **DB 파일은 앱데이터 폴더**(`app_data_dir`)에 둬야 앱 업데이트 후에도 유지됨.

### 크로스플랫폼 실무 메모
- **빌드는 OS별로 분리**: Mac에서 Windows 인스톨러 생성 불가(크로스컴파일 난이도 높음).
  → **GitHub Actions CI**로 Mac/Windows 빌드 자동화가 표준. (배포 시점 과제, 초기 개발엔 무관)
- **WebView 엔진**: Mac=WKWebView(내장), Windows=**WebView2**(런타임 필요).
  Tauri 인스톨러가 자동 설치/번들하도록 설정.
- 쿠팡 연동·크롤링용 Rust crate(`reqwest`/`hmac`/`sha2`/`rusqlite`)는 전부 크로스플랫폼 → OS 무관.
- **코드 서명**: Mac 공증(notarization) / Windows Authenticode — 배포 단계 과제(미서명 시 경고만).

## SQLite 동작 방식 (별도 설치가 필요 없는 이유)

> "로컬 SQLite를 사용자 PC에 어떻게 자동 설치하지?" → **설치하는 게 아니다.** SQLite는 서버가
> 아니라 라이브러리라서 앱 안에 같이 들어간다.

- **Postgres/Supabase**: 별도 서버 프로세스가 따로 돌아야 함 → 설치 필요.
- **SQLite**: 서버 아님. 그냥 코드 라이브러리. **빌드 시 앱 실행파일(`.exe`/`.app`)에 정적 링크되어
  내장됨.** 별도 프로세스·포트·설치 과정 없음. 사용자는 앱만 설치하면 SQLite가 이미 안에 있음.

### DB 파일은 첫 실행 때 자동 생성
DB **파일**(`.db`)은 빌드에 포함되지 않고, 앱을 처음 켤 때 생성된다.

```
앱 첫 실행
  → app_data_dir 에 data.db 있나?
  → 없으면: 파일 생성 + CREATE TABLE ... (마이그레이션) 실행
  → 있으면: 그대로 열어서 사용
```

- 테이블 생성 스크립트 = **마이그레이션(migration)**. 코드에 미리 정의해두면 첫 실행 시 자동 적용.
- 따라서 **빈 DB로 배포해도 사용자 PC에서 스키마가 알아서 만들어진다.**

### Tauri에서의 선택지
1. **`tauri-plugin-sql`** (공식): 마이그레이션 배열 정의 → 첫 실행 자동 적용. JS(프론트)에서 SQL 호출. 간단.
2. **`rusqlite`** crate: Rust 백엔드에서 직접 제어. 더 세밀함.

둘 다 SQLite C 라이브러리가 빌드 시 바이너리에 포함되므로 **사용자는 아무것도 설치하지 않는다.**

### 요약
| 시점 | 일어나는 일 |
|------|-------------|
| 빌드 | SQLite 엔진이 앱 바이너리에 내장 (자동) |
| 설치 | 사용자는 앱만 설치, SQLite 별도 설치 없음 |
| 첫 실행 | DB 파일 + 테이블 자동 생성 (마이그레이션) |
| 이후 실행 | 기존 DB 파일 열어서 사용 |

## 현재(Next.js) → Tauri 매핑 아이디어

| 현재 | Tauri에서 |
|------|-----------|
| API Route (`/api/admin/*`) | Rust `#[tauri::command]` 또는 sidecar |
| 쿠팡 HMAC 서명 (Node `crypto`) | Rust `hmac` + `sha2` crate |
| `CoupangClient.request` (fetch) | Rust `reqwest` |
| 크롤링 (cheerio) | Rust `scraper`/`reqwest`, 또는 webview |
| Supabase (Postgres) | **로컬 SQLite** (`tauri-plugin-sql`/`rusqlite`) — 확정 |
| `settings` 테이블 / secret key | **로컬 저장** (앱데이터/secure store) — 확정 |
| 쿠키 인증 / 미들웨어 | 로컬 앱이라 불필요 (네트워크 인증 자체가 사라짐) |
| 프론트(React/Next 페이지) | 그대로 React + Vite로 이식 가능 (Tailwind 유지) |

## 포팅 시 그대로 가져갈 수 있는 자산

- `src/lib/coupang/*` — "독립 모듈"로 설계됨. 로직(서명 메시지 구성, 엔드포인트, 상태 매핑)은 언어만 바꿔 1:1 이식 가능.
- `src/lib/crawler.ts` 의 셀렉터 맵 (05 문서) — 크롤링 규칙 그대로 사용.
- 상품 요청 JSON 구조 (06 문서) — 쿠팡 요청 스키마는 동일.

## 주의할 함정 (재현 필수)

- 쿠팡은 **HTTP 200이어도 `code:"ERROR"`** → 에러 처리 이중화.
- 응답 `data` 가 **숫자(sellerProductId)** 로 올 수 있음.
- 생성 vs 수정의 **contents 형식 차이** + 수정 시 `sellerProductItemId`/`vendorItemId` 주입 필요.
- 서명 datetime은 **UTC `yyMMddTHHmmssZ`**, 서명과 헤더 동일 값.
- 등록/수정 후 즉시 조회하면 상태 미반영 → 대기 후 조회.
